# Spec for league-invite-link

Title: League Invite Link with Confirmation Screen
Branch: claude/feature/league-invite-link
Spec file: context/specs/league-invite-link.md

## Summary

Allow league admins to share a direct invite URL (e.g. `/join/:inviteCode`) that anyone can click to join a league. Clicking the link shows a confirmation screen with the league name. If the user is not logged in, they are redirected to login or register and then returned to the confirmation screen automatically. If they are already logged in, the confirmation screen is shown immediately. The user must explicitly accept before being added to the league.

## Functional Requirements

- Add a `/join/:inviteCode` frontend route accessible to unauthenticated users.
- Add a public backend endpoint `GET /leagues/invite/:inviteCode` that returns the league name (and nothing else sensitive) without requiring authentication. Returns 404 if the code is invalid.
- On the confirmation screen, display the league name and two actions: "Join League" and "Decline" (or go back).
- If the user is not logged in when they visit `/join/:inviteCode`, redirect them to the login page. After successful login or registration, redirect back to `/join/:inviteCode` to complete the flow.
- If the user is already a member of the league, show an appropriate message instead of the join button (e.g. "You're already a member of this league").
- On successful join, redirect the user to the dashboard with the newly joined league set as active.
- In League Settings, add a "Copy invite link" button alongside the existing invite code display.
- The invite link uses the existing invite code — no new code generation is needed.

## Possible Edge Cases

- Invalid or expired invite code: show a clear error on the confirmation screen.
- User visits the link while already logged in and already a member: show "already a member" state.
- User visits the link while logged in as a user in another league: join succeeds, active league switches to the new one.
- User declines: stay on the confirmation screen or navigate back to wherever they came from.
- Login redirect loop: if the user logs out after being redirected to login, the invite code should be preserved so they can log back in and complete the flow.
- The invite code is regenerated by the league admin while a link is in-flight: the old link 404s and the confirmation screen shows a "link is no longer valid" message.

## Acceptance Criteria

- [ ] `GET /leagues/invite/:inviteCode` returns `{ leagueName }` without auth; returns 404 for unknown codes.
- [ ] `/join/:inviteCode` route renders a confirmation screen showing the league name.
- [ ] Unauthenticated users visiting `/join/:inviteCode` are redirected to login, then returned to the confirmation screen after auth.
- [ ] Authenticated users see the confirmation screen immediately.
- [ ] Clicking "Join League" calls `POST /leagues/join` and on success redirects to `/dashboard` with the league active.
- [ ] If the user is already a member, the join button is replaced with an "already a member" message.
- [ ] Invalid invite code shows a clear error (not a blank page or generic 404).
- [ ] League Settings section shows a "Copy invite link" button that copies the full URL to clipboard.

## Open Questions

- ~~Should the confirmation screen use the app shell or standalone layout?~~ **Decided: standalone full-page layout.** Simpler for unauthenticated users; no navbar auth dependency.
- ~~Should "Decline" navigate back in history, go home, or do nothing?~~ **Decided: navigate back in history.**

## Testing Guidelines

- Test the public `GET /leagues/invite/:inviteCode` endpoint: valid code returns league name, invalid code returns 404, no auth required.
- Test the frontend redirect: unauthenticated visit preserves the invite code through login and lands back on the confirmation screen.
- Test the already-a-member state.
- Test the invalid code error state on the confirmation screen.

## Personal Opinion

This is a straightforward and high-value feature — sharing a link is far more natural than copying and pasting an invite code. The existing invite code infrastructure handles everything on the backend; the work is almost entirely frontend routing and a small public endpoint.

The main thing to get right is the auth redirect loop: storing the `/join/:inviteCode` path in `localStorage` or as a `?redirect=` query param on the login URL so it survives the auth flow. This is a standard pattern and should be simple to implement cleanly.
